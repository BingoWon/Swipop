|-----|------|

## 常见问题

### Q1: 为什么不使用 React/Next.js？

**答**：这些框架需要编译过程，会带来以下问题：
1. 需要维护编译服务器
2. 编译可能失败，错误处理复杂
3. 用户需要等待编译完成
4. 增加服务器成本

我们的用户不是开发者，不应该关心任何技术细节。纯三件套无需编译，直接存储直接渲染。

### Q2: 不用框架能实现复杂效果吗？

**答**：可以。通过 CDN 引入的库可以实现绝大多数效果：
- Three.js：3D 效果
- D3.js：数据可视化
- GSAP：复杂动画
- Particles.js：粒子效果
- 等等...

这些库都可以通过一行 `<script>` 标签引入，无需 NPM。

### Q3: 为什么不让用户上传编译好的文件？

**答**：违背了产品理念。我们的目标用户不是开发者，他们不知道什么是编译，也不应该知道。

### Q4: 作品代码有安全风险吗？

**答**：WebView 提供了沙箱隔离，作品代码：
- 无法访问 iOS 原生功能（除非显式授权）
- 无法访问其他作品的数据
- 无法访问用户隐私数据

后续会增加代码审核机制。

### Q5: 为什么用按钮切换而不是滑动？

**答**：前端作品本身可能包含滑动交互（如滚动列表、游戏操作）。如果用滑动切换作品，会与作品内部的滑动冲突。独立的切换按钮避免了这个问题。

### Q6: 为什么选择 Supabase？

**答**：
1. 开箱即用，降低后端开发成本
2. 官方 Swift SDK
3. 内置 Google/Apple 登录
4. 免费额度适合 MVP
5. 可随时迁移到自建 PostgreSQL

### Q7: 为什么禁止 CocoaPods？

**答**：项目决策。Swift Package Manager 是苹果官方推荐的包管理方式，与 Xcode 集成更好，依赖管理更简洁。

### Q8: 作品的预览图怎么生成？

**答**：方案待定，可能的方式：
1. 服务端使用 Puppeteer 截图
2. 客户端 WebView 截图后上传
3. 用户手动上传

### Q9: 如何防止恶意代码？

**答**：多层防护：
1. WebView 沙箱隔离
2. 内容审核（人工 + 后续 AI）
3. 用户举报机制
4. 代码静态分析（后续）

### Q10: 为什么第一阶段不支持邮箱/手机号登录？

**答**：降低 MVP 复杂度：
- 邮箱需要发送验证邮件
- 手机号需要对接短信服务
- Google/Apple 登录已覆盖大部分用户
- 后续根据需求再添加

## 设计原则

1. **用户不是开发者**：所有技术复杂度由平台承担
2. **简单优先**：能简单实现的不复杂化
3. **借鉴成熟方案**：抖音的交互和社交已经过验证
4. **渐进增强**：先跑通核心流程，再逐步完善
