# 交互设计

## 核心交互问题

### 问题：滑动冲突

**矛盾点**：
- 抖音通过上下滑动切换视频
- 但前端作品本身可能包含上下滑动的交互（如滚动列表、游戏操作）
- 视频内部没有滑动事件捕获，但网页会捕获滑动事件

**结论**：无法沿用抖音的滑动切换方式

---

## 解决方案：独立切换按钮

### 设计思路

将作品切换操作独立出来，做成固定按钮，类似抖音左侧的点赞按钮。

### 按钮布局

```
┌─────────────────────────────────┐
│                                 │
│                                 │
│        [ 作品内容区域 ]          │  ← 作品可自由交互
│        (WebView 渲染)           │     不受切换按钮影响
│                                 │
│                           [▲]   │  ← 上一个作品
│                           [♥]   │  ← 点赞
│                           [★]   │  ← 收藏
│                           [💬]  │  ← 评论
│                           [➤]   │  ← 转发
│                           [▼]   │  ← 下一个作品
│                                 │
└─────────────────────────────────┘
```

### 按钮功能

| 按钮 | 功能 | 说明 |
|-----|------|-----|
| ▲ | 上一个作品 | 新增，抖音没有 |
| ♥ | 点赞 | 同抖音 |
| ★ | 收藏 | 同抖音 |
| 💬 | 评论 | 同抖音 |
| ➤ | 转发/分享 | 同抖音 |
| ▼ | 下一个作品 | 新增，抖音没有 |

### 设计优势

1. **无冲突**：作品内部的滑动操作不会触发作品切换
2. **明确**：用户清楚知道如何切换作品
3. **一致**：与点赞等按钮保持视觉一致性
4. **可扩展**：按钮可 hover 展开更多操作

---

## 悬浮交互

### 按钮 Hover 展开

当用户长按或悬停在按钮区域时，可展开显示更多操作入口：

```
普通状态          悬停/长按状态
   [▲]              [▲ 上一个]
   [♥]     →        [♥ 128 ]
   [★]              [★ 收藏 ]
   [💬]             [💬 32  ]
   [➤]              [➤ 分享 ]
   [▼]              [▼ 下一个]
```

---

## 作品内部交互

### 交互能力

作品（WebView）内部可实现任意交互：
- 点击、滑动、拖拽
- 键盘输入
- 陀螺仪/加速度计（需授权）
- 手势识别

### 交互隔离

- 作品区域的所有事件由 WebView 处理
- 原生按钮区域的事件由 iOS 处理
- 两者互不干扰

## 响应式设计优势

### 前端作品的自适应能力

| 设备类型 | 视频 | 前端作品 |
|---------|------|---------|
| 直板手机 | 固定比例 | ✅ 可适配 |
| 全面屏手机 | 黑边/裁切 | ✅ 可适配 |
| 折叠屏 | 无法适配 | ✅ 可适配 |
| iPad | 黑边 | ✅ 可适配 |
| 电脑横屏 | 黑边 | ✅ 可适配 |

**关键点**：前端作品通过 CSS 媒体查询可实现优雅的多设备适配，这是视频内容永远无法做到的。
